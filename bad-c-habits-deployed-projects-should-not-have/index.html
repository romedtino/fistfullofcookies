Bad C++ Habits Deployed Projects Should Not Have in 2017.

Exhibit A

using namespace inside a header file.

Example

<badheader.h>

pragma once

using namespace std;

class myheader {
...
}

Why is this bad?

Adding std is even worse but that's a whole different story. Look below.

#include "badheader.h"

struct string {
    const  char * p;
}

int main() {
    string banana; // problem!! which string? std::string? or struct string?
}

You'll run into a compiler error since it won't know which string you are trying to reference. Now imagine this inclusion is fifty layers deep. You can debug and find the root of this but you should have never been in this situation in the first place! Just don't do it!

Exhibit B

Creating SDKs or public header libraries that don't require a namespace path in the include statement.

Example

#include "MyAwesomeLibrary.h" //please don't!
instead use something like
#include "myAwesomeSauce/MyAwesomeLibrary.h" //better!

Why is this bad?

It is very similar to Exhibit A in nature. If you have a lot libraries in your project there's a decent chance the project has classes with the same names. For example in your project you have a file
[myproject]/include/LockGuard.h
and also a file
[myproject]/SpecialThreadLibrary/include/LockGuard.h

Now when you start adding include directories, it is pretty easy to end up saying
-I[myproject]/include
and
-I[myproject]/SpecialThreadLibrary/include

Then you start adding this in your program
<some.cpp>
include "LockGuard.h"

How does your compiler know which LockGuard.h you are referring to? This depends on the order of your include directories and usually it's first come, first serve. A good example is the Boost library in which you always do something like
#include "boost/algorithm/algorithm.hpp"
which helps prevent collisions such as this.

Exhibit C

Having multiple classes in one translation unit/CPP file. BONUS: Having your class have a TOTALLY different name than your translation unit.

Example

<Burger.h>

#pragma once
class cheese {
...
}

Why is this bad?

Compiling and running, it will work albeit less efficient. From a maintenance standpoint, this is a nightmare. Imagine having problems with class cheese and trying to figure out why you can't find cheese.cpp/h because it is actually under Burger.h. Will you remember that fact a year or two from now if you need to update class cheese? Also notice the lowercase of the class name. shudder. I am unsure who abused you as a child to end up writing code like this but we can get you help!

Conclusion

Honestly, I don't think the baseline I am toying with is that old but it's very interesting to see little nuances such as these. Don't get me wrong, the baseline is AMAZING. There's a lot of REALLY smart code in there. A lot of template magic, magical lambdas, and just modern C++ usage in general but it's very surprising to find some really sluggish looking code too. I guess this is just a downside of having multiple individuals/styles work on a large baseline. Regardless, if you're reading this, from this day forward, please don't do the things above! :D
